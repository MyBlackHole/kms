## 一、概述

### 1.1 什么是 KMS

密钥管理系统（Key Management Service, KMS）是一个用于安全创建、存储、管理、轮换和销毁加密密钥的集中式平台。它解决的核心问题是：

- **消除密钥散落**：将密钥从代码、配置文件、环境变量中剥离，统一收归管理
- **提供安全存储**：基于 HSM 等硬件可信根保护密钥材料
- **精细访问控制**：基于策略的权限模型，按最小权限原则控制密钥使用
- **全生命周期管理**：密钥生成 → 分发 → 使用 → 轮换 → 归档 → 销毁
- **审计追溯**：记录所有密钥操作，满足合规要求

### 1.2 核心术语

| 术语 | 全称 | 说明 |
|------|------|------|
| **CMK** | Customer Master Key | 客户主密钥，用户自主创建和管理的顶层密钥 |
| **KEK** | Key Encryption Key | 密钥加密密钥，用于加密 DEK 的二级密钥 |
| **DEK** | Data Encryption Key | 数据加密密钥，直接用于加密业务数据 |
| **HSM** | Hardware Security Module | 硬件安全模块，专用加密设备，密钥不出设备 |
| **Envelope Encryption** | 信封加密 | 用 KEK 加密 DEK，用 DEK 加密数据的分层加密模式 |
| **BYOK** | Bring Your Own Key | 客户导入自有密钥材料到 KMS |
| **KMIP** | Key Management Interoperability Protocol | OASIS 标准密钥管理互操作协议 |
| **PKCS#11** | Cryptoki | 密码令牌标准接口，HSM 通用 API |

## 二、核心技术原理

### 2.1 信封加密（Envelope Encryption）

这是 KMS **最核心的设计模式**，解决了性能与安全的根本矛盾。

#### 为什么需要信封加密？

1. **KMS API payload 限制**（AWS KMS 单次 Encrypt 上限 4KB）
2. **HSM 性能有限**（通常仅数百到数千 ops/sec），不适合大体积数据直接加密
3. **对称加密比非对称快几个数量级**（硬件加速 AES 可达数 GB/s）

#### 工作流程

```
加密流程：
┌──────┐    ① GenerateDataKey(CMK)     ┌─────┐
│ 应用 │ ────────────────────────────→  │ KMS │
│      │ ←── Plaintext DEK + Enc DEK ── │     │
│      │                                └─────┘
│      │ ② 用 Plaintext DEK 加密数据
│      │ ③ 存储 Enc DEK + 加密后的数据
│      │ ④ 销毁 Plaintext DEK
└──────┘

解密流程：
┌──────┐    ① Decrypt(Enc DEK)          ┌─────┐
│ 应用 │ ────────────────────────────→  │ KMS │
│      │ ←──── Plaintext DEK ────────── │     │
│      │                                └─────┘
│      │ ② 用 Plaintext DEK 解密数据
│      │ ③ 销毁 Plaintext DEK
└──────┘
```

#### 实现要点

- DEK 通常为 AES-256 对称密钥，每加密一次生成一个新的 DEK
- Plaintext DEK 用后即焚（从内存中安全清除）
- Encrypted DEK 与密文一起存储（可放在同一行/对象中）
- 解密时只需发送 Encrypted DEK（~1KB）到 KMS，而非整个数据

### 2.2 密钥分层体系

现代 KMS 普遍采用三级密钥分层：

```
┌─────────────────────────────────────┐
│   L1: Root Key（根密钥）            │  ← 存储在 HSM 内部
│   用途：保护 KEK                    │    永不以明文形式导出
│   算法：AES-256 / RSA-4096          │    由 HSM 物理防护
├─────────────────────────────────────┤
│   L2: KEK（密钥加密密钥）           │  ← 由 Root Key 加密存储
│   用途：保护 DEK                    │    每个部门/应用独享
│   算法：AES-256                     │    支持轮换版本
├─────────────────────────────────────┤
│   L3: DEK（数据加密密钥）           │  ← 由 KEK 加密存储
│   用途：直接加密业务数据            │    一次一密
│   算法：AES-256-GCM                 │    随密文持久化
└─────────────────────────────────────┘
```

**分层的好处**：

- Root Key 极少使用，可以放在最安全的 HSM 中
- KEK 可以按部门/应用隔离，泄漏洞一个不影响全局
- DEK 一次性使用，即使泄露也只影响单条数据

### 2.3 硬件安全模块（HSM）

HSM 是整个 KMS 的**信任根（Root of Trust）**和**可信计算基（TCB）**的核心。

#### HSM 关键特性

| 特性 | 说明 |
|------|------|
| **密钥不出设备** | 密钥在 HSM 内部生成，所有加密操作必须在设备内完成 |
| **物理防护** | 防篡改外壳，检测到入侵时自动零化（Zeroization）销毁密钥 |
| **真随机数生成器** | 基于物理熵源，符合 FIPS 标准 |
| **专用加密硬件** | 独立于通用 CPU，不受操作系统漏洞影响 |
| **认证标准** | FIPS 140-2 Level 3 / FIPS 140-3 |

#### HSM 部署形态

```
物理 HSM（PCIe卡/网络设备）
├─ 最高安全性（独立硬件、物理防护）
├─ 性能稳定（专用加密芯片）
├─ 合规首选（FIPS 140-2 L3 认证）
└─ 成本高，运维复杂

云 HSM（AWS CloudHSM / Azure Dedicated HSM）
├─ 便捷获取，弹性扩展
├─ 兼容 PKCS#11 / KMIP
├─ 物理隔离但运维托管
└─ 适合云原生部署

软件 HSM（SoftHSM）
├─ 开发测试环境使用
├─ 与 PKCS#11 应用兼容
├─ 无物理安全保护
└─ 不可用于生产
```

#### HSM 标准接口

| 接口 | 说明 | 适用场景 |
|------|------|---------|
| **PKCS#11 (Cryptoki)** | C API，最广泛的 HSM 接口 | 通用，跨平台 |
| **KMIP** | OASIS 标准网络协议 | 企业 KMS 间互操作 |
| **SDF** | 国密标准接口（GM/T 0018） | 国密 HSM |
| **SKF** | 智能密码钥匙接口（GM/T 0016） | 智能卡/UKey |

## 三、系统架构设计

### 3.1 总体架构：控制平面与数据平面分离

```
┌──────────────────────────────────────────────────────┐
│                     API Gateway                       │
│           路由 / 认证 / 限流 / TLS 终止               │
└──────────┬────────────────────────────────┬───────────┘
           │                                │
┌──────────▼──────────┐    ┌───────────────▼───────────┐
│   Control Plane     │    │       Data Plane          │
│   (控制平面)        │    │       (数据平面)          │
│                     │    │                          │
│  密钥生命周期管理   │    │  Encrypt / Decrypt       │
│  策略引擎(Policy)   │    │  Sign / Verify           │
│  审计日志管理       │    │  GenerateDataKey         │
│  用户/角色管理      │    │                          │
│  元数据存储         │    │  无状态，可水平扩展       │
│  (低频, 管理面)     │    │  (高频, 数据面)          │
└──────────┬──────────┘    └───────────────┬───────────┘
           │                                │
           └──────────┬─────────────────────┘
                      │
              ┌───────▼────────┐
              │   HSM Cluster  │
              │   (信任根)     │
              └────────────────┘
```

**设计原则**：

- 控制面与数据面独立扩展，互不影响
- 数据面无状态，可轻松水平扩展
- HSM 集群为共享资源池，支持负载均衡
- 元数据存储优先 CP（一致性优先于可用性）

### 3.2 密钥生命周期管理

```
         ┌──────┐
         │ 生成  │  ← HSM 内部生成 / 外部导入密钥材料
         └──┬───┘
            │
         ┌──▼───┐
         │ 启用  │  ← 新密钥默认启用
         └──┬───┘
            │
         ┌──▼───┐      ┌───────┐
         │ 使用  │ ←──→ │ 轮换  │  ← 生成新版本，旧版本保留解密能力
         └──┬───┘      └───────┘
            │
         ┌──▼───┐
         │ 禁用  │  ← 停止新加密，仍可解密已有数据
         └──┬───┘
            │
         ┌──▼───┐
         │ 归档  │  ← 仅保留解密能力，不可用于新加密
         └──┬───┘
            │
         ┌──▼───┐
         │ 销毁  │  ← 删除密钥材料，所有关联数据不可解密
         └──────┘
```

**关键设计决策**：

- 每个密钥有多个**版本**（key versions），轮换时创建新版本而非覆盖
- 旧版本保留用于解密历史数据，但不可用于新加密
- 销毁必须确认：设置等待期（如 7-30 天）防止误操作
- 禁用/归档操作需要特殊权限（多人审批）

### 3.3 访问控制模型

基于策略的权限控制（Policy-Based Access Control）：

```json
{
  "Version": "2026-06-01",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Role": "payment-service" },
      "Action": ["kms:Encrypt", "kms:Decrypt", "kms:GenerateDataKey"],
      "Resource": "kms:/keys/payment-key",
      "Condition": {
        "IpAddress": { "SourceIp": "10.0.0.0/8" },
        "DateAfter": { "AwsCurrentTime": "2026-06-01T00:00:00Z" }
      }
    },
    {
      "Effect": "Allow",
      "Principal": { "Role": "security-admin" },
      "Action": ["kms:CreateKey", "kms:RotateKey", "kms:DeleteKey"],
      "Resource": "*"
    }
  ]
}
```

**策略类型**：

- **Key Policy**：绑定到单个密钥的权限策略
- **IAM Policy**：基于用户/角色的全局权限
- **Grant**：临时授权，有时间范围和操作限制

### 3.4 审计日志

每笔操作必须记录：

| 字段 | 说明 | 示例 |
|------|------|------|
| `request_id` | 请求唯一标识 | `req_a1b2c3d4` |
| `timestamp` | 操作时间（毫秒精度） | `2026-06-24T10:30:00.123Z` |
| `principal` | 操作主体 | `user:payment-svc / role:admin` |
| `action` | 操作类型 | `kms:Decrypt` |
| `resource` | 操作对象 | `key:payment-key:v3` |
| `source_ip` | 客户端 IP | `10.0.1.100` |
| `result` | 结果（success/denied/error） | `success` |
| `signature` | 审计日志签名链 | `sha256(prev_hash + data)` |

## 四、主流 KMS 实现对比

### 4.1 HashiCorp Vault

| 维度 | 说明 |
|------|------|
| **语言** | Go |
| **许可证** | BSL（Business Source License，非纯开源） |
| **GitHub** | 30k+ stars |
| **存储后端** | Raft（内置）/ Consul / 文件系统 / 云存储 |
| **HSM 支持** | ✅ PKCS#11, KMIP, 云 KMS Auto-unseal |
| **KMIP** | ✅ Enterprise 版本支持 |

**核心引擎**：

| 引擎 | 说明 |
|------|------|
| **KV Secrets Engine** | 静态秘密存储（加密后写盘） |
| **Transit Engine** | 加密即服务（Encryption-as-a-Service） |
| **PKI Engine** | 证书颁发与管理 |
| **Database Engine** | 动态数据库凭据生成 |
| **Key Management Engine** | 企业版 KMS 密钥分发（Enterprise） |

**优势**：

- Auto-unseal：使用云 KMS/HSM 自动解封

- 许可证从 MPL 2.0 切换至 BSL，社区 fork 出 OpenBao

## 五、自建 KMS 技术选型

### 5.1 核心模块设计

```
┌─────────────────────────────────────────────────────┐
│                API Layer                             │
│     RESTful (OpenAPI) / gRPC / KMIP                 │
├─────────────────────────────────────────────────────┤
│            Authentication Layer                      │
│     mTLS / OIDC / JWT / API Token                   │
├─────────────────────────────────────────────────────┤
│              Policy Engine                           │
│     基于策略的访问控制（OPA / 自研）                │
├────────────────────────┬────────────────────────────┤
│  Key Lifecycle         │   Crypto Operations        │
│  生成 / 轮换           │  Encrypt / Decrypt         │
│  归档 / 销毁           │  Sign / Verify             │
│  版本管理              │  GenerateDataKey           │
├────────────────────────┴────────────────────────────┤
│            Storage Abstraction                       │
│    加密存储密钥材料到 DB / 文件系统                  │
├─────────────────────────────────────────────────────┤
│            HSM Abstraction Layer                     │
│     PKCS#11 | KMIP | Cloud KMS API                  │
├─────────────────────────────────────────────────────┤
│              Audit Logger                            │
│     结构化日志 + 签名链                              │
└─────────────────────────────────────────────────────┘
```

### 5.2 高可用设计

```
                    ┌──────────────┐
                    │  客户端应用   │
                    └──────┬───────┘
                           │ HTTPS (TLS 1.3)
                           ▼
                    ┌──────────────┐
                    │  反向代理     │  ← Nginx/HAProxy
                    │  TLS 终止     │
                    └──────┬───────┘
                           │ HTTP/1.1
                    ┌──────┴───────┐
                    │  KMS 服务     │
                    │  (主备部署)   │
                    └──────┬───────┘
                           │
               ┌───────────┼───────────┐
               ▼           ▼           ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │  主数据库  │ │  备用 HSM │ │  审计存储  │
        │ PostgreSQL│ │  PKCS#11  │ │  独立 DB   │
        └──────────┘ └──────────┘ └──────────┘
```

**设计要点**：

- KMS 实例无本地状态，所有持久状态在数据库和 HSM 中
- 密钥材料由 KEK 保护存储在数据库
- KEK 由共享 HSM 管理
- 支持水平扩展

### 5.3 技术选型参考

| 模块 | 推荐技术 | 理由 |
|------|---------|------|
| **API 框架** | Go (gin/fiber) / Rust (actix) | 性能、内存安全 |
| **认证** | OIDC + mTLS | 企业级 SSO，双向 TLS |
| **策略引擎** | OPA (Open Policy Agent) | 声明式策略，生产验证 |
| **元数据存储** | PostgreSQL / etcd | 强一致性，支持事务 |
| **加密库** | Go: aead / Rust: ring | 经安全审计的库 |
| **HSM 接口** | PKCS#11 (software: SoftHSM v2) | 标准接口，可与硬件 HSM 互换 |
| **审计存储** | 结构化日志 → 对象存储 / SIEM | 低成本，长周期保存 |
| **指标监控** | Prometheus + OpenTelemetry | 标准可观测性 |

### 5.4 演进路线

| 阶段 | 目标 | 具体做法 | 时间预估 |
|------|------|---------|---------|
| **Phase 1: MVP** | 消除密钥硬编码 | 部署 Vault/OpenBao，统一 API，接入核心应用 | 1-2 周 |
| **Phase 2: 服务化** | 完善生命周期 | 自动化轮换、策略引擎、审计日志、SDK 封装 | 1-2 月 |
| **Phase 3: 硬化** | 引入 HSM 信任根 | 集成云 HSM / 物理 HSM，迁移 Root Key | 1-2 月 |
| **Phase 4: 高可用** | 生产级 HA | 数据平面水平扩展、HSM 集群、跨 AZ 部署 | 2-4 周 |
| **Phase 5: 全球化** | 多 Region | 密钥仪式、跨区域同步、数据本地化合规 | 2-3 月 |

### 6.1 决策树

```
需要 KMS 能力？
```

## 七、结论

### 7.1 核心结论

1. **信封加密是 KMS 的核心模式**——平衡性能与安全的唯一正确方式
2. **HSM 是生产级 KMS 的信任根**——软件方案可用于 MVP，生产必须硬化
3. **控制面与数据面分离**——独立扩展，互不影响，是高可用的基础
4. **不要从零自研 KMS**——除非有极其特殊的需求。优先使用云厂商 KMS 或 Vault 等成熟方案
5. **渐进式演进**——从软件 KMS 开始，逐步引入国密 HSM、通过密评

### 7.2 风险提示

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| KMS 单点故障 | 所有服务不可用 | 多实例 + HSM 集群 + 本地缓存 |
| 密钥泄露 | 数据被解密 | HSM 保护 + 审计日志 + 快速轮换 |
| 合规不通过 | 无法上线 | 提前规划密评 + 使用认证产品 |
| 厂商锁定 | 迁移困难 | 标准接口（PKCS#11/KMIP） |
| 运维复杂 | 人力成本高 | 使用托管服务 / 培训团队 |
