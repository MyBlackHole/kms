## 一、KMS 存在的必要性

### 1.1 根本矛盾：密钥既要被使用，又要被保护

```
密钥的两难困境：
├── 使用密钥 → 必须让程序拿到明文 → 有泄露风险
└── 保护密钥 → 不能让任何人拿到 → 无法使用

KMS 的本质：在"可用"与"安全"之间找到平衡点
```

### 1.2 没有 KMS 时的世界

```
密钥散落在各处：
├── 源代码中: API_KEY = "sk-abc123..."      → git 仓库泄露
├── 配置文件中: password = "xxx"            → 运维人员可查看
├── 环境变量中: export DB_KEY=...           → 进程内存可 dump
├── 硬编码在App中: static KEY: [u8; 32]    → 逆向工程可提取
└── 传递在消息中: header: Bearer sk-xxx     → 中间人可截获

后果：
1. 一个密钥泄露 → 全面沦陷
2. 无法知道谁在什么时候用了什么密钥
3. 密钥轮换极其困难（需要改所有使用方）
4. 无法满足合规要求（等保、PCI DSS、GDPR）
```

### 1.3 KMS 的本质定义

> **KMS 是一个"密钥的银行"——你把密钥存进去，需要时来取，用完就走，全程有监控。**

| 银行类比 | KMS 对应 |
|---------|---------|
| 存钱进银行 | 密钥加密存储在 KMS |
| 取款需要身份验证 | API 调用需要认证（Token/mTLS） |
| 银行有监控摄像头 | 审计日志记录每次操作 |
| 银行有保险柜 | HSM 硬件保护根密钥 |
| 可以挂失银行卡 | 密钥可以禁用/销毁 |

## 二、KMS 解决的五大问题

### 2.1 密钥散落（Key Sprawl）

**症状**：密钥出现在代码、配置、环境变量、文档、聊天记录中

**KMS 如何解决**：

```
Before: App 直接持有密钥明文
┌──────┐     ┌──────┐
│ App  │────▶│ DB   │  密钥在 App 代码/配置中
└──────┘     └──────┘

After: App 只持有密文 DEK，需要时向 KMS 请求
┌──────┐ ①请求DEK  ┌─────┐
│ App  │──────────▶│ KMS │  App 没有密钥明文
│      │◀──────────│     │  只有被加密的 DEK
│      │ ②返回密文DEK└─────┘
│      │
│      │ ③用DEK本地加解密  ┌──────┐
│      │──────────────────▶│ DB   │
└──────┘                  └──────┘
```

**效果**：密钥只存在于 KMS 内部，应用代码中没有任何密钥明文

### 2.2 密钥泄露后无法止损（No Revocation）

**症状**：密钥泄露了，但你不知道哪些系统在用，改一个地方漏一个地方

**KMS 如何解决**：

```
密钥状态机：
┌─────────┐    禁用     ┌──────────┐    归档    ┌──────────────┐
│ Enabled │────────────▶│ Disabled │──────────▶│ PendingArchive│
└────┬────┘             └──────────┘           └──────┬───────┘
     │                                                │ 销毁
     │ 轮换                                           ▼
     ▼                                         ┌──────────┐
┌─────────┐                                    │ Destroyed│
│ Enabled │  (新版本)                           └──────────┘
│ v2      │
└─────────┘

紧急止损流程：
1. 发现泄露 → 立即 disable 该密钥
2. 所有用该密钥加密的数据 → 无法继续加密新数据
3. 已有密文 → 仍可用旧版本 DEK 解密（不影响历史数据）
4. 排查完毕 → 轮换到新版本，重新加密受影响数据
5. 确认安全 → 销毁泄露的旧版本
```

**效果**：一键禁用，全网生效，无需改任何应用代码

### 2.3 密钥轮换极其困难（Key Rotation Nightmare）

**症状**：密钥用了 3 年没换过，因为换了要改 50 个服务的配置

**KMS 如何解决**：

```
版本化密钥管理：

密钥 prod-db-key:
├── v1 (2024-01-01) → 状态: Disabled  → 用于解密历史数据
├── v2 (2024-06-01) → 状态: Disabled  → 用于解密历史数据
├── v3 (2026-01-01) → 状态: Disabled  → 用于解密历史数据
└── v4 (2026-06-01) → 状态: Enabled   → 当前活跃版本

轮换流程：
1. KMS 自动生成新版本 v5
2. 新请求用 v5 加密
3. 旧请求仍可用 v4 解密
4. 无需改任何应用代码
5. 达到保留上限(如10个) → 最旧版本自动淘汰
```

**效果**：密钥轮换从"改 50 个服务"变成"调一个 API"

### 2.4 无法审计谁用了什么密钥（No Audit Trail）

**症状**：出了安全事故，无法追溯是谁在什么时候用了哪个密钥

**KMS 如何解决**：

```
审计日志哈希链：

Event1 ──SM3──▶ hash1
  │
  ▼
Event2 ──SM3──▶ hash2 (prev_hash = hash1)
  │
  ▼
Event3 ──SM3──▶ hash3 (prev_hash = hash2)

每条记录包含：
├── 谁 (subject): "user:payment-svc"
├── 做了什么 (action): "kms:Decrypt"
├── 对哪个密钥 (resource): "key:prod-db-key:v3"
├── 什么时候 (timestamp): "2026-06-25T10:30:00Z"
├── 来自哪里 (source_ip): "10.0.1.100"
├── 结果 (result): "success" / "denied"
└── 签名 (signature): SM2 签名防篡改

防篡改原理：
- 修改任何一条记录 → SM3 哈希改变 → 链断裂
- 攻击者无法在不留痕迹的情况下篡改审计日志
```

**效果**：完整的操作追溯链，满足等保三级审计要求

### 2.5 合规要求（Compliance Requirements）

**症状**：等保、PCI DSS、GDPR 都要求密钥管理，但不知道怎么做

#### 等保（网络安全等级保护）概述

等保是对信息系统进行分等级安全保护的制度，依据 **GB/T 22239-2019《网络安全等级保护基本要求》**。

**五个等级的定义**：

| 等级 | 名称 | 适用对象 | 受侵害客体 | 受侵害程度 |
|------|------|---------|-----------|-----------|
| **一级** | 用户自主保护级 | 一般系统 | 公民、法人和其他组织的合法权益 | 一般损害 |
| **二级** | 系统审计保护级 | 一般系统 | 公民、法人和其他组织的合法权益；社会秩序和公共利益 | 一般损害 or 严重损害 |
| **三级** | 安全标记保护级 | 重要系统 | 公民、法人和其他组织的合法权益；社会秩序和公共利益；国家安全 | 严重损害 or 特别严重损害 |
| **四级** | 结构化保护级 | 关键基础设施 | 社会秩序和公共利益；国家安全 | 特别严重损害 |
| **五级** | 访问验证保护级 | 极端重要系统 | 国家安全 | 特别严重损害 |

**等级保护流程**：

```
定级 → 备案 → 建设整改 → 等级测评 → 监督检查

1. 定级：由运营单位自定级别，专家评审，主管部门审核
2. 备案：到当地公安机关备案
3. 建设整改：按相应级别要求进行安全建设和整改
4. 等级测评：由具备资质的测评机构进行测评
5. 监督检查：公安机关定期检查
```

#### 等保各级别对 KMS 的具体要求

##### 一级（用户自主保护级）— 基本身份鉴别

| 控制点 | 要求 | KMS 实现 |
|--------|------|---------|
| 身份鉴别 | 应对登录用户进行身份标识和鉴别 | 基本用户名/密码认证 |
| 访问控制 | 应对登录用户分配账户和权限 | 简单的用户角色划分 |
| 安全审计 | 应启用安全审计功能 | 基础操作日志记录 |
| 数据完整性 | 应采用校验技术保证重要数据完整性 | SM3 哈希校验 |
| 数据保密性 | 应采用加密或其他保护措施实现重要数据保密性 | SM4 加密存储 |

**KMS 要求**：基本的密钥存储和加解密功能，无 HSM 要求，审计日志可选。

##### 二级（系统审计保护级）— 审计与访问控制

| 控制点 | 要求 | KMS 实现 |
|--------|------|---------|
| 身份鉴别 | 应采用两种或两种以上组合鉴别用户身份 | 双因素认证（密码+短信/令牌） |
| 访问控制 | 应根据管理用户角色分配权限，实现管理用户的权限分离 | RBAC 角色权限控制 |
| 安全审计 | 审计记录应包括事件日期时间、主体、客体、结果等 | 结构化审计日志 |
| 数据完整性 | 应采用校验技术或密码技术保证重要数据完整性 | SM3 哈希 + HMAC |
| 数据保密性 | 应采用密码技术保证重要数据保密性 | SM4-GCM 加密 |
| 密码技术 | 应使用国家密码管理局认可的密码算法和产品 | SM2/SM3/SM4 国密算法 |

**KMS 要求**：国密算法支持，RBAC 权限控制，结构化审计日志，可使用软件密码模块。

##### 三级（安全标记保护级）— HSM 与审计防篡改

| 控制点 | 要求 | KMS 实现 |
|--------|------|---------|
| 身份鉴别 | 应采用两种或两种以上组合鉴别用户身份；应对远程管理通道进行加密 | mTLS + Token 双向认证 |
| 访问控制 | 应对重要主体和客体进行安全标记；应对重要操作进行权限分离 | ABAC 策略引擎 + 安全标记 |
| 安全审计 | 应启用安全审计功能，审计覆盖每个用户；审计记录应不可删除修改 | SM3 哈希链 + SM2 签名防篡改 |
| 入侵防范 | 应能检测到重要节点的位置并进行监控 | 实时告警 + 入侵检测 |
| 数据完整性 | 应采用密码技术保证重要数据在传输和存储过程中的完整性 | SM3 HMAC + SM2 签名 |
| 数据保密性 | 应采用密码技术保证重要数据在传输和存储过程中的保密性 | SM4-GCM + 信封加密 |
| 数据备份 | 应提供重要数据的本地数据备份与恢复功能 | 备份 + 异地容灾 |
| 剩余信息保护 | 应保证鉴别信息所在的存储空间被释放前得到完全清除 | zeroize 安全擦除 |
| 密码技术 | 应使用国家密码管理局认可的密码算法和产品 | 国密算法 + 国密 HSM |

**KMS 要求**：
- **必须使用 HSM**：根密钥必须存储在国密 HSM 中
- **审计防篡改**：SM3 哈希链 + SM2 签名，审计日志不可删除修改
- **密钥全生命周期**：生成→分发→使用→轮换→归档→销毁
- **mTLS 通信加密**：所有 API 通信必须使用国密 TLS

##### 四级（结构化保护级）— 高可用与灾备

| 控制点 | 要求 | KMS 实现 |
|--------|------|---------|
| 身份鉴别 | 应采用密码技术保证重要身份鉴别信息的保密性 | SM2 证书 + 智能密码钥匙 |
| 访问控制 | 应对重要主体和客体进行安全标记并实施强制访问控制 | MAC 强制访问控制 |
| 安全审计 | 应对审计进程进行保护；审计日志应实时传送 | 独立审计服务器 + 实时传输 |
| 入侵防范 | 应能发现已知漏洞并及时修补；应能检测到用户行为异常 | 漏洞管理 + 行为分析 |
| 恶意代码防范 | 应采用主动免疫可信验证机制 | 可信计算平台 |
| 数据完整性 | 应采用密码技术保证重要数据在传输和存储过程中的完整性 | 端到端签名 |
| 数据保密性 | 应采用密码技术保证重要数据在传输和存储过程中的保密性 | 端到端加密 |
| 数据备份 | 应提供重要数据异地实时备份功能 | 异地双活 + 实时同步 |
| 剩余信息保护 | 应保证鉴别信息所在的存储空间被释放前得到完全清除；应保证存有敏感数据的存储空间被释放或重新分配前得到完全清除 | 内存加密 + 安全擦除 |
| 密码技术 | 应使用国家密码管理局认可的密码算法和产品；密码设备应具有密码产品认证证书 | 国密 HSM（三级认证） |

**KMS 要求**：
- **HSM 三级认证**：密码设备必须通过国密三级认证
- **异地灾备**：跨地域实时备份，支持故障自动切换
- **可信计算**：引入可信验证机制，防止内部人员作案
- **实时审计**：审计日志实时传送到独立存储

##### 五级（访问验证保护级）— 最高安全

| 控制点 | 要求 | KMS 实现 |
|--------|------|---------|
| 物理安全 | 应选择国家级安全机房；应采取电磁屏蔽措施 | 国家级保密机房 + 电磁屏蔽 |
| 网络安全 | 应采用国密 VPN 建立专用网络通道 | 国密 IPSec/SSL VPN |
| 身份鉴别 | 应采用生物特征 + 智能密码钥匙 + PIN 三因素认证 | 多因素强认证 |
| 访问控制 | 应基于安全标记实施强制访问控制；应实现三权分立 | MAC + 三权分立（系统管理/安全保密/安全审计） |
| 安全审计 | 审计日志应有独立存储；应实施审计分析和违规检测 | 独立审计系统 + AI 分析 |
| 数据安全 | 应采用密码技术保证数据全生命周期安全 | 端到端加密 + 密钥托管 |

**KMS 要求**：国家密码管理部门审批，专用密码设备，三权分立，电磁屏蔽机房。

#### 等保各级别 KMS 要求对比总结

| 维度 | 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|------|
| **认证方式** | 用户名/密码 | 双因素 | mTLS + Token | SM2 证书 + 智能钥匙 | 三因素 |
| **权限控制** | 简单角色 | RBAC | ABAC + 安全标记 | MAC 强制访问控制 | 三权分立 |
| **审计要求** | 基础日志 | 结构化日志 | 哈希链防篡改 | 独立审计服务器 | AI 分析 |
| **密码算法** | 可选 | 国密 | 国密 | 国密 | 国密 |
| **HSM 要求** | 无 | 可选 | 必须（二级认证） | 必须（三级认证） | 必须（最高认证） |
| **通信加密** | 可选 | TLS | 国密 TLS (GMTLS) | 国密 VPN | 专用网络 |
| **数据备份** | 本地 | 本地 | 异地容灾 | 异地实时 | 国家级灾备 |
| **密钥生命周期** | 可选 | 基本管理 | 全生命周期 | 全生命周期 + 审计 | 全生命周期 + 托管 |

#### 密评（密码应用安全性评估）

密评依据 **GB/T 39786-2021《信息安全技术 信息系统密码应用基本要求》**，是对信息系统密码应用合规性的专项评估。

**密评测评维度**：

| 维度 | 测评内容 | 达标方式 |
|------|---------|---------|
| **物理与环境安全** | 密码设备应放置在受控机房 | HSM/密码机部署于标准机房 |
| **网络与通信安全** | 传输信道采用国密 SSL/TLS | GMTLS（国密 TLS）/ IPSec |
| **设备和计算安全** | 使用密码技术实现身份鉴别 | SM2 证书 + 智能密码钥匙 |
| **应用和数据安全** | 应用层数据加密 + 完整性校验 | KMS SDK 对接 SM4-GCM + SM3 |
| **密钥管理安全** | 密钥全生命周期管理 | KMS 系统覆盖所有密钥管理环节 |
| **管理制度** | 密钥管理制度 + 人员职责 | 配套管理制度文档 |

**密评评分标准**：

```
密评总分 = 各维度得分 × 权重

达标要求：
├── 每个维度得分 ≥ 60 分
├── 总分 ≥ 70 分
└── 无高风险项

高风险项（一票否决）：
├── 使用未经认证的密码产品
├── 密钥材料明文存储
├── 审计日志可被删除修改
└── 未使用国密算法（等保要求时）
```

#### 国际合规标准对比

| 标准 | 地区 | 核心要求 | 与 KMS 的关系 |
|------|------|---------|-------------|
| **等保二级/三级** | 中国 | 国密算法、HSM、审计防篡改 | 必须使用 SM2/SM3/SM4 |
| **FIPS 140-2/3** | 美国 | HSM 认证、加密库认证 | AWS KMS、Azure HSM 已认证 |
| **PCI DSS** | 全球 | 密钥至少每年轮换、访问控制、审计 | 信封加密 + 自动轮换 |
| **GDPR** | 欧盟 | 个人数据加密保护 | 信封加密保护数据 |
| **SOC 2** | 全球 | 密钥管理控制证据 | 审计日志 + 访问控制 |

## 三、KMS 本身带来的问题

KMS 不是银弹，它解决了上述五大问题的同时，引入了新的问题：

### 3.1 单点故障（Single Point of Failure）

```
KMS 挂了 → 所有依赖它的服务都无法加解密

典型故障场景：
├── KMS 服务器宕机
├── HSM 硬件故障
├── 网络分区（KMS 在另一个机房）
├── 数据库不可用（存储密钥元数据）
└── 证书过期（TLS 通信失败）

缓解方案：
├── 多实例部署 + 负载均衡
├── HSM 集群（Active-Active）
├── 跨 AZ 部署
├── 本地缓存 DEK（短暂离线可用）
└── 灾备方案（密钥仪式恢复）
```

### 3.2 性能瓶颈（Performance Bottleneck）

```
每次加解密都要调 KMS API → 网络延迟

延迟对比：
├── 本地 AES 加密: ~1μs
├── 调用 KMS API:  ~5-50ms (网络往返)
└── 差距: 5000-50000 倍

信封加密的权衡：
├── 第一次: 调 KMS 生成 DEK → 5-50ms
├── 后续: 本地用 DEK 加密 → ~1μs
└── 权衡: 用一次网络开销换取后续无限本地性能

DEK 缓存的权衡：
├── 不缓存: 每次加密都调 KMS → 性能差
├── 缓存 1-5 分钟: 大幅提升性能
└── 权衡: 缓存期间 DEK 暴露在应用内存中
```

### 3.3 运维复杂度（Operational Complexity）

```
KMS 引入的运维负担：
├── HSM 硬件管理
│   ├── 采购周期长（数月）
│   ├── 需要专业机房环境
│   └── 固件升级、故障替换
├── 密钥仪式（Key Ceremony）
│   ├── 根密钥生成需要多人在场
│   ├── 密钥分片物理运输
│   └── 灾备恢复流程复杂
├── 密评（密码应用安全性评估）
│   ├── 需要通过第三方测评
│   ├── 年度复评
│   └── 整改成本高
└── 人员要求
    ├── 需要密码学专业人员
    ├── 需要安全运维团队
    └── 培训成本高
```

### 3.4 迁移成本（Migration Cost）

```
从"无 KMS"到"有 KMS"的迁移：
├── 需要改造所有使用密钥的应用
├── 需要重新加密历史数据
├── 需要培训开发和运维团队
├── 需要编写 SDK 封装 KMS 调用
└── 需要灰度验证，不能一刀切

时间估算：
├── 小型系统（<10 个服务）: 1-2 月
├── 中型系统（10-50 个服务）: 3-6 月
└── 大型系统（>50 个服务）: 6-12 月
```

### 3.5 信任转移（Trust Transfer）

```
KMS 的本质是把信任从"散落的密钥"转移到"KMS 系统本身"

新的信任链：
├── 你信任 KMS 服务 → KMS 挂了你怎么办？
├── 你信任 HSM 硬件 → HSM 厂商有后门怎么办？
├── 你信任运维人员 → 内部人员作案怎么办？
├── 你信任网络通信 → TLS 被中间人攻击怎么办？
└── 你信任审计日志 → 审计系统被入侵怎么办？

没有完美的安全，只有权衡：
KMS 把"密钥散落"的 100 个风险
收敛为"KMS 本身"的 5 个风险
→ 这 5 个风险可以用成熟方案（HSM、mTLS、RBAC）加固
```

## 四、各实现方案对比

### 4.1 云厂商 KMS（AWS KMS / GCP Cloud KMS / 腾讯云 KMS）

```
解决的问题：
├── 零运维：云厂商管理所有基础设施
├── 即开即用：几分钟内可用
├── 合规认证：云厂商已通过 FIPS 140-2/3
├── 生态集成：与云服务无缝集成（S3、RDS 等）
└── 高可用：云厂商保证 SLA

引入的新问题：
├── 厂商锁定：迁移到另一个云需要重新设计
├── 成本：按 API 调用计费，大规模使用费用高
├── 数据主权：密钥材料在云厂商的 HSM 中
├── 限流：单次 API 调用有 payload 限制（AWS KMS 4KB）
└── 延迟：跨区域调用延迟高
```

**适用场景**：单一云厂商、不想自运维、非核心系统

### 4.2 HashiCorp Vault / OpenBao

```
解决的问题：
├── 多云支持：不绑定特定云厂商
├── 功能丰富：不只是 KMS，还有秘密管理、PKI、动态凭据
├── 插件生态：多种存储后端、认证方式、审计后端
├── 开源可控：OpenBao 是纯开源（MPL 2.0）
└── Auto-unseal：用云 KMS/HSM 自动解封

引入的新问题：
├── 复杂度高：配置项多，学习曲线陡峭
├── 性能有限：默认配置性能不高，需要调优
├── 运维负担：需要自己部署、监控、升级
├── 许可证风险：Vault 从 MPL 切换到 BSL（OpenBao 解决）
└── 密评难度：需要自行通过密码应用安全性评估
```

**适用场景**：混合云/多云、需要秘密管理、有运维团队

### 4.3 自建 KMS

```
解决的问题：
├── 完全自主可控：代码自己写，随时可改
├── 国密合规：SM2/SM3/SM4 原生支持
├── 定制能力强：可以做任何需要的功能
├── 无厂商锁定：不依赖任何第三方
└── 成本可控：长期来看比云 KMS 便宜

引入的新问题：
├── 开发成本高：需要密码学专业人员
├── 安全责任自负：没有第三方认证背书
├── 密评难度大：需要自证合规
├── 维护成本高：需要持续更新、修复漏洞
├── 功能不完善：很多功能是桩代码，需要补齐
└── 风险高：密码学实现错误可能导致严重后果
```

**适用场景**：核心系统、极高安全要求、信创环境、有专业团队

### 4.4 国产商用 KMS 产品（数字认证/数盾科技/密码云芯）

```
解决的问题：
├── 开箱即用：产品已经过密评认证
├── 国密原生：SM2/SM3/SM4 + 国密 HSM
├── 厂商支持：有专业团队实施和运维
├── 信创适配：适配国产芯片和操作系统
└── 合规保障：产品已有认证，过密评更容易

引入的新问题：
├── 成本高：商业授权费用
├── 厂商锁定：绑定特定厂商的 HSM 和软件
├── 定制受限：功能由厂商定义，定制能力有限
├── 升级依赖：需要厂商配合升级
└── 供应商风险：厂商倒闭或停止支持
```

**适用场景**：预算充足、需要快速过密评、信创环境

## 六、核心结论

1. **KMS 的本质**：把"密钥散落"的 100 个风险，收敛为"KMS 本身"的 5 个可控风险

2. **信封加密是核心模式**：KMS 只管密钥（控制面），数据加密在本地（数据面），平衡了性能与安全

3. **没有免费的午餐**：每个方案都在"运维成本 / 自主可控 / 合规难度 / 功能丰富度"之间做权衡

4. **自建 KMS 的代价**：最大的问题不是开发，而是**密评**——你需要自证合规，这比写代码难得多

5. **务实建议**：除非有极其特殊的需求（如国密合规 + 信创环境 + 核心系统），否则优先使用成熟方案（云 KMS 或 Vault）

## 附录：核心术语表

### HSM（Hardware Security Module，硬件安全模块）

**定义**：一种专用的密码学硬件设备，用于安全地生成、存储和管理加密密钥。

**核心特性**：

| 特性 | 说明 |
|------|------|
| **密钥不出设备** | 密钥在 HSM 内部生成，所有加密操作必须在设备内完成，密钥材料永不以明文形式导出 |
| **物理防篡改** | 防篡改外壳，检测到物理入侵时自动零化（Zeroization）销毁所有密钥 |
| **真随机数生成器** | 基于物理熵源（如热噪声、电子噪声）生成真随机数，符合 FIPS 标准 |
| **专用加密硬件** | 独立于通用 CPU 运行，不受操作系统漏洞影响 |
| **认证标准** | 通过 FIPS 140-2 Level 3 / FIPS 140-3 认证（国际）或国密二级/三级认证（国内） |

**在 KMS 中的角色**：HSM 是整个 KMS 的**信任根（Root of Trust）**。根密钥（Root Key）存储在 HSM 内部，永不离开设备边界，所有密钥加密操作都在 HSM 内完成。

**部署形态**：

```
物理 HSM（PCIe 卡 / 网络设备）
├── 最高安全性（独立硬件、物理防护）
├── 性能稳定（专用加密芯片）
├── 合规首选（FIPS 140-2 L3 认证）
└── 成本高，运维复杂

云 HSM（AWS CloudHSM / Azure Dedicated HSM）
├── 便捷获取，弹性扩展
├── 兼容 PKCS#11 / KMIP
├── 物理隔离但运维托管
└── 适合云原生部署

软件 HSM（SoftHSM）
├── 开发测试环境使用
├── 与 PKCS#11 应用兼容
├── 无物理安全保护
└── 不可用于生产
```

**标准接口**：

| 接口 | 说明 | 适用场景 |
|------|------|---------|
| **PKCS#11 (Cryptoki)** | C API，最广泛的 HSM 接口标准 | 通用，跨平台 |
| **SDF** | 国密标准接口（GM/T 0018） | 国密 HSM |
| **SKF** | 智能密码钥匙接口（GM/T 0016） | 智能卡/UKey |
| **KMIP** | OASIS 标准网络协议 | 企业 KMS 间互操作 |

### DEK（Data Encryption Key，数据加密密钥）

**定义**：直接用于加密业务数据的密钥，是密钥分层体系中的最底层密钥。

**在信封加密中的角色**：

```
密钥分层体系：
┌─────────────────────────────────────┐
│   Root Key（根密钥）                 │  ← 存储在 HSM 内部
│   用途：保护 KEK                     │    永不以明文形式导出
├─────────────────────────────────────┤
│   KEK（Key Encryption Key）          │  ← 由 Root Key 加密保护
│   用途：保护 DEK                     │    按部门/应用隔离
├─────────────────────────────────────┤
│   DEK（Data Encryption Key）         │  ← 由 KEK 加密保护
│   用途：直接加密业务数据              │    一次一密，用完即销毁
└─────────────────────────────────────┘
```

**DEK 的生命周期**：

```
生成 → 使用 → 销毁

1. 生成：KMS 生成随机 DEK（如 32 字节），用 KEK 加密后返回密文 DEK
2. 使用：应用用明文 DEK 在本地加密数据（SM4-GCM / AES-256-GCM）
3. 存储：密文 DEK 与密文数据一起存储
4. 解密：应用发送密文 DEK 给 KMS，KMS 用 KEK 解密返回明文 DEK
5. 销毁：应用用完 DEK 后立即从内存中安全清除（zeroize）
```

**DEK 的特点**：

| 特点 | 说明 |
|------|------|
| **一次一密** | 每次加密操作生成新的 DEK，即使泄露也只影响单条数据 |
| **本地加密** | 数据加密在应用本地完成，不经过网络，性能无损 |
| **用后即焚** | 明文 DEK 使用后立即从内存中安全清除 |
| **密文存储** | DEK 以密文形式（被 KEK 加密）与数据一起存储 |
| **无需网络** | 加解密操作不依赖 KMS 服务，KMS 不成为性能瓶颈 |

**DEK 与 KEK 的关系**：

```
加密数据时：
1. KMS 生成 DEK → 用 KEK 加密 DEK → 返回密文 DEK
2. 应用用明文 DEK 加密数据 → 得到密文数据
3. 存储：密文 DEK + 密文数据

解密数据时：
1. 应用发送密文 DEK 给 KMS
2. KMS 用 KEK 解密密文 DEK → 返回明文 DEK
3. 应用用明文 DEK 解密密文数据 → 得到明文数据
```
